推理模式 ¶

c10::InferenceMode 是一个类似于 NoGradMode 的新 RAII 保护，用于当你确定你的操作将与 autograd 无交互时（例如模型训练）。与 NoGradMode 相比，在此模式下运行的代码通过禁用与 autograd 相关的工作（如视图跟踪和版本计数器更新）可以获得更好的性能。然而，在 c10::InferenceMode 内部创建的张量在与 autograd 系统交互时也有更多的限制。

InferenceMode 可以用于代码块。在 InferenceMode 内部，所有新分配的（非视图）张量都被标记为推理张量。推理张量：

  • 由于没有版本计数器，所以如果尝试读取它们的版本（例如，因为保存了这个张量以供反向传播）将会引发错误。

  • 在 InferenceMode 之外是不可变的。因此，如果尝试以下操作将会引发错误：- 在 InferenceMode 之外修改它们的数据。- 在 InferenceMode 之外将它们修改为 requires_grad=True 。为了解决这个问题，可以在 InferenceMode 之外创建一个克隆以获得在修改之前的正常张量。

一个非视图张量只有在它是在 InferenceMode 内部分配的情况下才是推理张量。一个视图张量只有在它所视图的张量是推理张量的情况下才是推理张量。

在 InferenceMode 块内部，我们提供以下性能保证：

  • 如 NoGradMode 所述，所有操作即使输入有 grad_fn 也不会记录 requires_grad=True 。这适用于推理张量和普通张量。

  • 推理张量的视图操作不会进行视图跟踪。视图和非视图推理张量无法区分。

  • 推理张量的就地操作保证不会进行版本升级。

更多关于 InferenceMode 的实现细节，请参阅 RFC-0011-InferenceMode。

迁移指南从 AutoNonVariableTypeMode ¶

在生产环境中使用 PyTorch 进行推理工作负载时，我们看到了 C++ 守卫 AutoNonVariableTypeMode （现在为 AutoDispatchBelowADInplaceOrView ）的广泛应用，该守卫禁用 autograd、视图跟踪和版本计数器增加。不幸的是，当前对推理工作负载中此守卫的口语化描述是不安全的：使用 AutoNonVariableTypeMode 可能绕过 PyTorch 的安全检查，导致静默错误的结果，例如，当为反向操作保存的张量随后被修改时，PyTorch 会抛出错误，但修改发生在 AutoNonVariableTypeMode 内部将静默绕过检查并返回错误的梯度给用户。

当当前 AutoNonVariableTypeMode 的用户考虑迁移时，以下步骤可能有助于您决定最佳替代方案：

  1. 尝试在仅推理模式下运行工作负载的用户（如加载预训练的 JIT 模型并在 C++ 运行时中进行推理）应在张量上的所有操作（包括模型加载）上添加 c10::InferenceMode guard 守卫。以下是一个推理工作负载的示例：

c10::InferenceMode guard;
model.load_jit(saved_model);
auto inputs = preprocess_tensors(data);
auto out = model.forward(inputs);
auto outputs = postprocess_tensors(out);

Note c10::InferenceMode 提供了 AutoNonVariableTypeMode 的直接替换方案，同时保留了 AutoNonVariableTypeMode 的性能特性。但它们也有一些差异，用户应该特别注意：

  • 两个守卫都会影响张量执行过程以跳过与推理无关的工作，但 InferenceMode 也会影响张量的创建，而 AutoNonVariableTypeMode 则不会。换句话说，在 InferenceMode 内创建的张量会被标记为推理张量，以便在退出 InferenceMode 后应用某些限制。

  • InferenceMode 的启用/禁用状态可以嵌套，而 AutoNonVariableTypeMode 只允许启用状态。

{
  InferenceMode guard(true);
  // InferenceMode is on
  {
    InferenceMode guard(false);
    // InferenceMode is off
  }
  // InferenceMode is on
}
// InferenceMode is off

  1. 尝试实现自定义内核并希望在 Autograd 分发键下重新分发的用户应使用 AutoDispatchBelowADInplaceOrView 。注意 AutoDispatchBelowADInplaceOrView 只是 AutoNonVariableTypeMode 的新名称，因为它更好地解释了守卫的功能。我们正在弃用 AutoNonVariableTypeMode ，并在 1.10 版本中将其删除。请参阅 pytorch/vision 中的自定义内核 ROIAlignFunction 的示例：

class ROIAlignFunction : public torch::autograd::Function<ROIAlignFunction> {
 public:
  static torch::autograd::variable_list forward(
      torch::autograd::AutogradContext* ctx,
      const torch::autograd::Variable& input,
      const torch::autograd::Variable& rois,
      double spatial_scale,
      int64_t pooled_height,
      int64_t pooled_width,
      int64_t sampling_ratio,
      bool aligned) {
    ctx->saved_data["spatial_scale"] = spatial_scale;
    ctx->saved_data["pooled_height"] = pooled_height;
    ctx->saved_data["pooled_width"] = pooled_width;
    ctx->saved_data["sampling_ratio"] = sampling_ratio;
    ctx->saved_data["aligned"] = aligned;
    ctx->saved_data["input_shape"] = input.sizes();
    ctx->save_for_backward({rois});
    // Used to be at::AutoNonVariableTypeMode g;
    at::AutoDispatchBelowADInplaceOrView guard;
    auto result = roi_align(
        input, rois, spatial_scale, pooled_height,
        pooled_width, sampling_ratio, aligned);
    return {result};
  }

自定义原地 & 视图内核需要除上述守卫之外的特殊处理，请参阅自定义内核教程以获取更多详细信息。
